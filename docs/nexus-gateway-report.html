📌背景と動機

スマートビルディングの設備は、BACnet・OPC-UA・MQTT・Modbus など複数のプロトコルで動作し、 各々が独自のアドレス体系・データ型・意味論を持つ。上位の Building OS は (gateway_id, point_id) を鍵とする単一の telemetry/control 契約を要求する。 このギャップをエッジで吸収する層として nexus-gateway は設計された。 docs/background.ja.md

主目的
プロトコル多様性の吸収
System of Record
Building OS（ゲートウェイではない）
ゲートウェイの責務
接続と変換のみ
ステータス
v0.1.0 public preview / MVP baseline: OPC-UA

なぜ EdgeX Foundry を使わないのか

EdgeX Foundry の Core Metadata（Device/Profile レジストリ）と Core Command（REST → Device Service）は、 Building OS がすでに Digital Twin として所有する責務と重複する。最小構成でも 10〜20 コンテナになる。 nexus-gateway はプロトコル別コンテナ分離と Common Event パイプラインの思想を「プラットフォームの重さなしに」借用している。 README.ja.md §EdgeX

🔭概要・コンセプト

nexus-gateway は「Building OS を System of Record とするための、エッジ側プロトコル吸収層」である。 機器・ポイント・デジタルツインの正本を Building OS が持ち、ゲートウェイは接続と変換だけを担う。 docs/background.ja.md §1

成功の定義

  • 新プロトコルのコネクタを OCI コンテナとしてデプロイするだけで対応できる（Core への変更不要）
  • BACnet・OPC-UA・MQTT 由来のテレメトリが、統一された common-model フォームで Building OS に届く
  • アップリンク停止中も SQLite バッファでデータを保持し、復旧後に順序を保ってフォワード
出典: docs/vision/vision.md

スコープ外（明示的に除外）

  • デバイスレジストリ・メタデータレジストリ・コマンド権限 → Building OS が所有
  • Kubernetes オーケストレーション（コンテナ型だが K8s 非依存）
  • セマンティック推論・知識グラフ・推論エンジン
  • マルチゲートウェイ連携（サイトをまたいだフェデレーション）
  • ファームウェア OTA（コネクタ OTA は IN スコープ、ファームウェアは除外）
出典: docs/vision/vision.md, MVP_READINESS.md

主要な設計決定（ADR）

ADR決定内容根拠
ADR-0001 コネクタはネイティブアドレスのみ発行、local_id → point_id は Normalizer が所有 Point List 変更時に Normalizer だけ変更すればよい。コネクタが canonical ID を持つと再 normalization が不可能
ADR-0002 テレメトリは best-effort。SQLite 有界リングバッファ、overflow 時 drop-oldest Building OS gRPC 契約が cumulative ack のみ（per-frame ID なし）のため exactly-once は実装不可能
ADR-0003 Point List の正本は Building OS twin。ゲートウェイは diff 同期の copy のみ 二重管理を避ける。ゲートウェイが Point List を編集すると twin との不整合が生じる
ADR-0004 制御は real-time-or-fail。core NATS request-reply（JetStream 不使用）、stale write 禁止 古い制御指令が障害復旧後に物理設備へ適用されると安全上の問題となる
ADR-0005 JetStream を Normalizer の前段に置く（raw event を replay 可能に） Point List 変更後に raw event を再 normalize できる。正規化済み telemetry だけだと修正不可能
ADR-0006 コネクタは署名済み OCI、digest 固定実行、Catalog 経由 cosign 検証 コネクタは物理設備への接続を持つため、unsigned image 実行はサプライチェーン攻撃リスク
ADR-0007 gRPC は Building OS Traefik エッジで mTLS 終端。gateway_id = cert CN/SAN 機械間通信は OIDC より mTLS が適切。Building OS 側は Traefik + cert-manager で実装済み（BOS #296/#298）

🔄データフロー / コネクタ

テレメトリ上り経路（Telemetry Uplink）

BACnet COV リアルタイムパス

BACnet COV 通知はポーリングと並列で動作し、同一の JetStream パイプラインを通じて Building OS に届く。 ゲートウェイ側での追加バッファリングはなく、COV 発火から BOS 到達まで通常 50 ms 未満（最悪 500 ms）。

ステージレイテンシ備考
_cov_loop → js.publish()< 5 ms非同期 publish、バッチなし
NATS JetStream キューイング< 1 msローカルプロセス間
Normalizer pull Fetch0–500 msmaxWait=500ms。負荷時は即時返却
S&F Buffer.Write + WriteNotify< 2 msSQLite WAL モード
Forwarder.drain → grpcSink.Send< 5 msWriteNotify で即時起動、送信側バッチなし
gRPC client-streaming → BOSネットワーク RTTCheckpoint（ACK 収集）は送信をブロックしない

制御下り経路（Control Downlink）

コネクタ契約（全プロトコル共通）

コネクタが実装すべき最小インターフェース。Building OS ドメインモデルを持たず、ネイティブアドレスのみ扱う。 

// NATS subject: evt.<protocol>.<connector_id>
subscribe() → Stream[CommonEvent]   // poll / COV subscribe

// NATS subject: cmd.<protocol>.<connector_id> (core NATS, not JetStream)
write(cmd: ControlCommand) → ControlResult  // stale deadline は即時エラー

経路方針理由

スケール試験結果（実機シミュレータ + 実ビル OS, 2026-06）

実シミュレータ（BACnet device 1001 / OPC-UA opc.tcp://192.0.2.10:4840）から 実ビル OS（192.0.2.20、ingress 5051 / egress 5052、平文）まで end-to-end で検証： sim → コネクタ → NATS JetStream（evt.<proto>.<connector>）→ gateway 正規化 → BOS。 完全性は BOS 側で受信した distinct point_id 数、スループットは gateway の store-and-forward メトリクス（/metrics）で測定。 connector/bacnet, connector/opcua, internal/storefwd

100 / 100100（Good）100（Good）00

2000 点（BACnet 1000 + OPC-UA 1000）が実ビル OS まで到達することを両側で確認。 BOS 側で 約 2000 distinct point_id を受信、テレメトリ（ingress 5051）・ 制御下り（egress 5052、GatewayEgress 接続成立、Unimplemented=0）とも有効。 normalizer_unresolved_total=0（全点プロビジョニング済み）。
50 分連続（BACnet 1000 @30s poll + OPC-UA 1000）で持続性を確認。 書込 ~26 frames/s に対し送信 ~31 frames/s（送信が書込を常に上回り backlog 発散なし）、 store-and-forward WAL は 1.7–4.7 MB で循環（有界）、gateway メモリ 14–26 MiB（リークなし）。 既知の限界：高書込レート（5s poll ≈ 200 writes/s）では store-and-forward SQLite が SQLITE_BUSY（database is locked） でドレイン律速 → busy_timeout / WAL チューニングが必要 （#109）。30s poll では 3–4 回/5 分で自己回復し持続可能。
本試験で顕在化・修正したスケール課題: ① BACnet ReadPropertyMultiple の APDU 超過 → BACNET_RPM_CHUNK_SIZE（既定 20）でチャンク分割。 ② bacpypes3 の read 無タイムアウトによる poll ループのハング → BACNET_READ_TIMEOUT（既定 5s）。 ③ 大規模時の COV 購読ストーム回避用に BACNET_COV_ENABLED（既定 true、poll-only 切替）。 ④ スケールフィクスチャを単一デバイス・distinctへ再生成（5 コネクタが同一 200 アドレスを再利用していた欠陥を解消）。 connector/bacnet, fixtures/scale/gen_master.py, docs/connector-spec.md §BACnet env

指数バックオフ（BOS 再接続）

internal/retry/backoff.go に実装。Min=1s, Max=60s, Factor=2.0, ±20% jitter。 BOS 障害時の再接続ストームを防ぐ。 internal/retry/backoff.go

1. gRPC mTLS（ADR-0007）

Building OS 側の Traefik エッジが mTLS を終端する（TLSOption: RequireAndVerifyClientCert + passTLSClientCert）。 証明書の CN/SAN が gateway_id に対応し、cert-manager が発行。Traefik が X-Gateway-Id ヘッダを注入し、 Building OS がフレームの gateway_id と一致強制（BOS #296/#298 で実装済み）。 GW 側のコード変更なし。設定追加のみで対応可。 docs/adr/0007-transport-security-mtls-at-edge.md §Update 2026-06-16 
# 本番設定例
GATEWAY_ID=GW-SOS-001
BOS_ADDR=bos.example.com:443
BOS_CA_FILE=/etc/nexus/tls/ca.pem
BOS_CERT_FILE=/etc/nexus/tls/gateway.crt  # CN/SAN = GATEWAY_ID
BOS_KEY_FILE=/etc/nexus/tls/gateway.key

2. コネクタ配布セキュリティ（ADR-0006）

Core Agent は unsigned image を実行しない / digest 不一致を実行しない / allowlist 外レジストリからプルしない。 これは強制されており、設定で緩和できない。GHCR を MVP レジストリとし、Harbor/ECR/ACR も対応予定。 docs/adr/0006-connector-distribution-signed-oci.md

3. Admin API / UI 認証

Keycloak OIDC。operator（書き込み）と viewer（読み取り）ロールを分離。 JWKS URL 未設定時は認証スキップ（dev/CI 専用）。 internal/adminapi/, SECURITY.md

スコープ外（脆弱性対象外）

  • compose スタックの dev 認証情報（operator/operator 等）— 明示的なプレースホルダー
  • --bos-insecure h2c — dev/CI の意図的な設定。本番デプロイでの使用は設定ミス
  • Building OS・Keycloak・NATS・Docker 自体の脆弱性 — それぞれの upstream へ
  • ホスト root 権限取得後の攻撃

アーキテクチャ提言

  1. Point List provisioning API を最優先外部依頼事項とする — ADR-0003 の「Building OS が正本」という設計は、この API なしには file/CSV に依存し続ける。 BOS #224 のデリバリーリスクを独立してトラッキング。
  2. BACnet / OPC-UA auto-discovery を早期に実装する（Issue #90, #91） — BOS 主導のポイント管理を実現するために必須。bacnet-sim-gateway 側は bows/acquire.py に実装済み、OPC-UA connector 側は Browse 結果の転用で完成。
  3. Connector SDK / manifest schema / conformance test を公開仕様化する — Connector Catalog が新たな中央集権的ロックインにならないよう、外部ベンダがコネクタを書けるエコシステムを整備。
  4. Control command の E5 シナリオを CI の必須ゲートに組み込む — stale write 禁止の保証は評価テスト E5 で検証予定だが、リグレッション防止のために CI に常時走らせることを推奨。
  5. BOS との gRPC 契約の互換性ポリシーを明文化する — gatewaybridge proto の breaking change policy がないと、BOS アップグレード時にすべての GW が一斉に壊れるリスクがある。

既知の限界事項

以下は現時点で確認されている実装上の制約であり、設計の失敗ではなく既知の MVP スコープ判断。
  • BACnet host networking は Docker Compose の制約上 CI ゲート外（ローカル実行では動作）
  • cosign 本番検証は MVP+1。現 MVP は dev/file catalog モード
  • Grafana ダッシュボード等の完全 observability スタックは MVP+1。/metrics テキストで代替
  • マルチゲートウェイ連携（サイトをまたいだフェデレーション）は現在の設計スコープ外
